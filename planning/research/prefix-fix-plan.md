# FSD Skill Prefix Duplication — Diagnosis & Fix Plan

**Date:** 2026-04-25  
**Symptom:** Every FSD skill appears twice in Claude Code's skill list — once with `fsd:` prefix and once without.  
**Severity:** UX friction; skills still work but the doubled list is confusing and unreliable.

---

## 1. Diagnosis

### 1.1 How Claude Code loads skills

Claude Code uses two separate registry files to locate plugin content:

| File | Key field | What it means |
|---|---|---|
| `~/.claude/plugins/installed_plugins.json` | `installPath` | Primary: the directory Claude Code loads skills FROM |
| `~/.claude/plugins/known_marketplaces.json` | `installLocation` | Secondary: marketplace root, scanned for plugin manifests |

For every other plugin, `installLocation` points at a neutral marketplace index directory (`~/.claude/plugins/marketplaces/<publisher>/`), not at the plugin itself.

### 1.2 What install.sh does differently for FSD

`install.sh` registers the FSD plugin via **both** mechanisms:

```
installed_plugins.json
  fsd@tbeack.installPath  →  ~/.claude/plugins/cache/tbeack/fsd/0.14.0/
                              (versioned copy of plugin/, synced at install time)

known_marketplaces.json
  tbeack.installLocation  →  ~/.claude/plugins/fsd/
                              (the live git clone root)
  tbeack.source.path      →  ~/.claude/plugins/fsd/
```

The clone root contains `.claude-plugin/marketplace.json` with `"source": "./plugin"`. Claude Code treats the clone root as a marketplace, discovers the plugin there, and also reads skills from `~/.claude/plugins/fsd/plugin/`.

**Result: Claude Code loads skills from two separate directory trees simultaneously.**

```
Path A (installPath → cache):
  ~/.claude/plugins/cache/tbeack/fsd/0.14.0/skills/*/SKILL.md
  ← synced at last install.sh run

Path B (installLocation → clone → plugin/):
  ~/.claude/plugins/fsd/plugin/skills/*/SKILL.md
  ← whatever the live git clone contains
```

### 1.3 Why names diverge

The two paths produce different `name:` values whenever either:

- The cache is stale (skills were modified or committed since the last `install.sh` run), OR
- The local working tree has uncommitted SKILL.md edits that only exist on disk and not in the clone (which pulls from GitHub remote)

**Current state:**
- 16 SKILL.md files have unstaged local modifications adding the `fsd:` prefix to their `name:` fields
- These changes have not been committed or pushed
- The git clone at `~/.claude/plugins/fsd` is at origin/main (`f95a736`) which has bare `name:` values (e.g., `name: spec`)
- The cache at `~/.claude/plugins/cache/tbeack/fsd/0.14.0/` was synced from the clone → also bare names
- Result: Path A and Path B both serve bare names; the duplication is coming from both paths being read, not from name differences in this particular moment

The underlying architecture guarantees future divergence any time:
1. SKILL.md names change in a commit without re-running `install.sh`
2. Local edits to SKILL.md exist on disk but are not pushed

### 1.4 Session startup alignment false signal

The FSD session hook reports `16 skills source↔cache aligned` by comparing the working tree (`/Users/theobeack/Desktop/AI/fsd/plugin/skills/`) to the cache. It does not verify whether Claude Code is loading from one path or two, and does not compare against what Claude Code actually sees.

## 2. Alternatives

### Option A — Re-run `install.sh` (symptom fix only)

Re-running install.sh syncs the cache from the clone and updates the metadata. If both paths end up with identical skill names, Claude Code may deduplicate and the symptom disappears.

**Pros:** Zero code change; fixes the immediate stale-cache case.  
**Cons:** Does not eliminate dual-path loading; the bug will recur on the next commit + cache drift. Requires remembering to re-run install.sh after every `name:` change. Does not fix the architecture.

**Verdict:** Not recommended as a standalone fix.

---

### Option B — Eliminate the marketplace registration (remove `tbeack` entry)

Remove the `tbeack` entry from `known_marketplaces.json`. This stops Claude Code from treating the clone as a secondary skill source. All loading goes through `installPath` (the cache) only.

**Pros:** Removes dual-path loading entirely; clean and simple.  
**Cons:** Still relies on a versioned cache that must be re-synced on every version change. The "marketplace" mechanic that resolves `CLAUDE_PLUGIN_ROOT` may break hooks that rely on it. Removing an entry from `known_marketplaces.json` may be regenerated by Claude Code on next update.

**Verdict:** Viable but brittle — the marketplace entry may be re-created, and the stale-cache problem persists.

---

### Option C — Point `installPath` at the live clone's `plugin/` directory (recommended)

Change `installed_plugins.json` so `installPath = ~/.claude/plugins/fsd/plugin` (the actual plugin subdirectory inside the git clone). Eliminate the versioned cache entirely.

```
Before:
  installPath → ~/.claude/plugins/cache/tbeack/fsd/0.14.0/   (stale copy)
  installLocation → ~/.claude/plugins/fsd/                    (live clone, also read)

After:
  installPath → ~/.claude/plugins/fsd/plugin/                 (live clone subdir, single source)
  installLocation → ~/.claude/plugins/fsd/                    (still present but installPath
                                                               already points into plugin/, so
                                                               no new content is added)
```

Under this scheme, `install.sh` still git-pulls the clone — but no cache copy is needed, because `installPath` reads directly from the clone. There is only one source of truth.

**Pros:**
- Eliminates the staleness problem at the root — no copy to go stale
- Eliminates the need to sync a cache on version bumps
- `install.sh` becomes simpler (no `cp -r` step, no stale-cache cleanup)
- Single directory serves as both `installPath` and the marketplace-discovered source

**Cons:**
- The versioned cache is how Claude Code tracks which version of a plugin is active; removing it may affect Claude Code's internal version tracking. Needs verification.
- If the `installLocation` path is still registered AND Claude Code still reads from it, there could still be dual-loading — but both paths now resolve to the same files, so duplicates would be deduplicated by name.

**Verdict:** Recommended. Even if Claude Code reads both paths, they now point to the same content.

---

### Option D — Separate the marketplace registry from the plugin content

Create a proper marketplace index at `~/.claude/plugins/marketplaces/tbeack/` that only contains discovery metadata (a `marketplace.json` pointing at the plugin's GitHub repo), separate from the plugin content itself. Update `known_marketplaces.json` to point `installLocation` at the index, not the plugin.

This mirrors how `claude-plugins-official` is structured.

**Pros:** Architecturally correct and matches the Claude Code plugin system design.  
**Cons:** Requires understanding the exact schema Claude Code expects for a local-directory marketplace index, which is undocumented. Risky to break the registration mechanism.

**Verdict:** Ideal long-term but higher risk given undocumented schema.

---

## 3. Recommended Approach: Option C

**Why Option C wins:**

The core invariant we need is: *there is exactly one directory from which Claude Code reads FSD skills*. Option C achieves this by making `installPath` point at the live clone's `plugin/` subdir — the same files that the marketplace registration also resolves to. Even if Claude Code reads from both registrations, they resolve to identical content, so any deduplication by name yields a clean list.

Option C also eliminates the stale-cache class of bugs permanently, not just for this occurrence.

## 4. Implementation Plan

### Step 1 — Update `install.sh`

Remove the versioned cache copy step (lines 90–112) and change the `installed_plugins.json` write to use `~/.claude/plugins/fsd/plugin` as `installPath`.

**Before (lines 95–111):**
```bash
VERSION=$(node -e "console.log(require('$CLONE_DIR/plugin/.claude-plugin/plugin.json').version)")
CACHE_DIR="$CACHE_BASE/$VERSION"

if [ -d "$CACHE_BASE" ]; then
  for old_dir in "$CACHE_BASE"/*/; do
    ...
    rm -rf "$old_dir"
  done
fi

mkdir -p "$CACHE_DIR"
cp -r "$CLONE_DIR/plugin/." "$CACHE_DIR/"
ok "Plugin content synced to $CACHE_DIR"
```

**After:**
```bash
VERSION=$(node -e "console.log(require('$CLONE_DIR/plugin/.claude-plugin/plugin.json').version)")
PLUGIN_DIR="$CLONE_DIR/plugin"   # live clone, no copy needed
ok "Plugin source: $PLUGIN_DIR"
```

**Before (lines 125–139) — `installed_plugins.json` write:**
```javascript
installPath: '$CACHE_DIR',
```

**After:**
```javascript
installPath: '$PLUGIN_DIR',
```

Remove `CACHE_BASE` from the script entirely (no longer needed).

### Step 2 — Clean up the existing stale cache

```bash
rm -rf ~/.claude/plugins/cache/tbeack/
```

This prevents Claude Code from loading the stale cache path if it persists in any session state.

### Step 3 — Commit local SKILL.md changes before re-running install

The 16 unstaged `name: fsd:*` edits need to be committed and pushed so that `install.sh`'s git pull brings them into the clone. Otherwise the clone remains at f95a736 with bare names.

Decision: verify whether the `fsd:` prefix in `name:` fields is intentional (it is the current design per FSD-018 session notes). If yes, commit, push, then re-run install.

### Step 4 — Re-run install.sh

```bash
bash install.sh
```

This will git-pull the clone (now including the `fsd:` prefixed names), and write `installPath = ~/.claude/plugins/fsd/plugin` to `installed_plugins.json`.

### Step 5 — Restart Claude Code and verify

Open a new session. The FSD session hook should report 16 skills. The system-reminder skills list should show each FSD skill exactly once, with `fsd:` prefix.

### Step 6 — Update session startup hook to validate single-path loading

Enhance `fsd-context-monitor.js` (or the session startup hook) to check that `installed_plugins.json`'s `installPath` for `fsd@tbeack` matches the expected live path. If a stale cache path is detected, emit a warning.

## 6. Risk Assessment

| Risk | Likelihood | Mitigation |
|---|---|---|
| Claude Code re-creates the cache on plugin re-registration | Low | `installPath` pointing at a real dir should be sufficient |
| `CLAUDE_PLUGIN_ROOT` env var breaks hook scripts | Low | Clone's `plugin/` is structurally identical to what was in the cache; no path changes for scripts inside the plugin |
| Duplicate registration via `installLocation` still causes doubles | Low | Both paths now resolve to same files; Claude Code deduplicates by `name:` |
| Future `git pull` brings changes that break running session | Acceptable | Same behavior as current cache-based system; session restart picks up changes |
